# HTML Guidelines

## Когда использовать этот скилл

- Создание компонентов с разметкой
- Вёрстка страниц и секций
- Ревью / исправление accessibility
- Работа с формами
- Оптимизация SEO

## Общие принципы

### Семантика

- Выбирать тег по смыслу, не по внешнему виду
- `<div>` и `<span>` — только когда нет подходящего семантического тега
- Один `<main>` на страницу
- Один `<h1>` на страницу, заголовки по порядку (h1 → h2 → h3)

### Доступность

- Интерактивные элементы доступны с клавиатуры
- Фокус всегда видимый (не убирать outline без замены)
- Изображения с информацией — информативный alt
- Декоративные изображения — `alt=""` или `aria-hidden="true"`
- Формы: каждый input связан с label

### Визуальные состояния

При кастомной стилизации интерактивных элементов — обязательно реализовать:

- `hover` — наведение курсора
- `focus` / `focus-visible` — фокус с клавиатуры
- `active` — момент клика/нажатия
- `disabled` — неактивное состояние
- `loading` — состояние загрузки (если применимо)
- `error` — состояние ошибки (для полей форм)

### SEO

- Заголовки отражают структуру контента
- Ссылки с информативным текстом (не "тут", "подробнее")
- Изображения с осмысленным alt
- Внешние/рекламные ссылки — `rel="nofollow"` при необходимости

## UI-библиотеки

При использовании компонентов из UI-библиотек (MUI, Semantic UI, Chakra, Ant Design и др.):

### Что учитывать

- Библиотека может автоматически добавлять ARIA-атрибуты, обработку фокуса, состояния
- Не дублировать то, что уже работает из коробки
- Использовать пропсы компонента вместо нативных атрибутов, где это предусмотрено
- Сверяться с документацией библиотеки

### Порядок действий

1. Проверить, есть ли готовый компонент в библиотеке
2. Изучить его API и встроенное поведение
3. Добавить только то, что библиотека не покрывает
4. При кастомизации стилей — сохранить все состояния

## Схема описания элемента

Каждый файл в `elements/` описывает элемент по структуре:

1. **Сценарии использования** — когда применять, когда нет
2. **Связанные элементы** — обязательные и опциональные связи
3. **Необходимые атрибуты** — функциональные, a11y, идентификация
4. **Визуальные состояния** — если кастомный или переопределяем стили
5. **Правила доступности** — клавиатура, screen reader, ARIA
6. **SEO** — если влияет на индексацию
7. **Пример** — эталонный код
